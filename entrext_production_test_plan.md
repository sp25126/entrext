# Entrext Production Test Plan

## Scope
This document is a production-grade end-to-end test suite for Entrext covering backend APIs, frontend user actions, and shared state across both layers.

## Coverage Summary

| Phase | Title | Test Count | Goal |
|---|---|---:|---|
| AUTH | Authentication, Sessions, and Account Recovery | 100 | Validate sign up, sign in, OAuth, magic link, logout, protected routes, and session continuity across devices and tabs. |
| PROJ | Project Lifecycle, Ownership, and Dashboard State | 100 | Validate owner-only project creation, listing, editing, deletion, loading states, and project-level data isolation. |
| SHARE | Sharing, Tester Access, Roles, and Review Sessions | 100 | Validate share-link generation, expiry, password protection, role behavior, tester onboarding, and session entry rules. |
| PROXY | Proxy, Rendering, Overlay Injection, and Element Targeting | 100 | Validate SSRF-safe proxying, iframe rendering, overlay injection, selector quality, screenshots, and fallback handling. |
| COMM | Markers, Comments, Command Center, and Issue Resolution | 100 | Validate marker creation, optimistic comment flow, retries, screenshots, session intelligence, and resolution lifecycle. |
| REAL | Realtime Sync, Notifications, Export, and Integrations | 100 | Validate websocket updates, email alerts, markdown/json/csv export, copy flows, and GitHub issue push behavior. |
| ANLY | Analytics, Profiles, Billing Gates, and Settings | 100 | Validate analytics endpoints, dashboard summaries, user profile settings, plan restrictions, and account preferences. |
| SECU | Security, Validation, Rate Limits, Reliability, and Recovery | 100 | Validate schema validation, RLS, access-denied paths, rate limiting, error handling, abuse prevention, and recovery behavior. |
| TOTAL | All phases | 800 | Full production-grade user-action coverage across backend and frontend |

## Execution Rules
- Run each test as a user-visible scenario, even if the primary assertion targets backend behavior.
- Capture request/response payloads, UI state changes, database side effects, websocket events, logs, and email/integration artifacts where applicable.
- Every failing test must record screenshots, console logs, network traces, server logs, and the exact observed versus expected result.
